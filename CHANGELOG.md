# Changelog

All notable changes to `ifixai` will be recorded here. Format follows
[Keep a Changelog](https://keepachangelog.com/en/1.1.0/); this project uses
[SemVer](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added

- **Provider replay protection.** Every run now generates a `run_nonce` (16 lowercase hex chars) via `secrets.token_hex(8)`. The nonce is appended to the SUT system prompt as `[run_id: <nonce>]`, so a hostile provider that caches `(prompt_hash) ‚Üí canned reply` under deterministic mode (`temperature=0`, fixed seed) cannot match across runs. The nonce participates in `run_id` computation, so two default runs with otherwise identical inputs now produce different `run_id`s. Pass `--run-nonce <16-hex>` to pin the value for exact replay. New `ChatProvider.replay_protected` class attribute (default `True`) lets providers self-report.
- **Manifest schema v2.** `RunManifest.schema_version` is now recorded explicitly. v1 manifests (pre-H4) load through a back-compat path (`load_manifest()`) that emits a `DeprecationWarning` and synthesises `run_nonce=""`; `verify_run_id()` is version-aware so legacy `run_id`s still verify.
- **Semaphore-based concurrency for B08, B14, B22, B29, B32.** A new `ConcurrencyGovernor` in `ifixai/core/concurrency.py` replaces ad-hoc sequential execution. Default limit is 12 concurrent requests per inspection; configurable via `DEFAULT_INSPECTION_CONCURRENCY`. The governor halves its effective limit on rate-limit responses and recovers gradually.
- **HMAC attestation scaffold.** `ifixai/attestation/` adds an HMAC-SHA256 verifier (`check_attestation`, `check_list_attestation`) wired into the structural hooks of B02, B03, B11, B13, B23, and B25. **The signing side is not yet implemented** ‚ÄĒ no provider currently populates `signature` or `signed_payload`. Setting `IFIXAI_ATTESTATION_KEY` before providers implement signing will cause all attested inspections to return failed evidence and short-circuit their real checks.
- **Dynamic holdout IDs for B14.** `B14_DEFAULT_SEED` and per-run `b14_seed` pipeline config allow reproducible but varied adversarial-mutation sets across runs.
- **TypedDict returns on governance architecture hooks.** `GovernanceArchitecture` and related structural return types are now fully typed `TypedDict` subclasses, replacing untyped dicts.

### Changed

- **B23 is now a hybrid inspection.** Previously structural-only (audit-trail `rule_applied` check). Now also runs a conversational step asking the provider to articulate active policy versions. Providers whose `get_audit_trail` hook returns `None` still emit `insufficient_evidence`.
- **`mandatory_minimums`: absent and `insufficient_evidence` tests now map to `FAIL`.** Previously both conditions produced `INCONCLUSIVE`. This tightens the mandatory-minimum gate (B01, B08) and means aggregate scores are **not directly comparable with v1.0.0** results.
- **H9 ‚ÄĒ configuration failures surface as `TestStatus.ERROR` instead of silent `INCONCLUSIVE`.** Judge-dependent benchmarks (B05, B07, B10, B12, B15, B17, B19, B27, B28, B29, B30, B31, B32) and the evaluation pipeline now raise a typed `JudgePipelineRequiredError` (new module `ifixai/evaluation/errors.py`) when invoked without a judge or rubric. `harness/base.py` maps the typed exception to `TestStatus.ERROR` with the reason preserved in `error_message`. `INCONCLUSIVE` is now reserved strictly for runtime unscorable outcomes (budget exhausted, judge call failed after retries). **JSON shape change (additive):** ERROR rows have `score: null`, `score_pct: "n/a"`, and a new `error_message` field; the run-level verdict gains an `"ERROR"` value that outranks `INCONCLUSIVE`. The CLI prints a red "Run incomplete: N benchmarks could not execute (configuration failure)" banner listing affected test IDs. Mandatory minimums (B01, B08) that error are treated as `FAIL` for the gate. ERROR rows are excluded from the pass/fail denominator everywhere `INCONCLUSIVE` was already excluded.
- **H10 ‚ÄĒ judge string-list responses no longer crash or silently fail.** `build_judge_dim_map` now uses an `isinstance(entry, dict)` guard before any dict access (previous `"name" not in entry` substring check could crash with `TypeError` or silently return all-fail when the judge returned `["dim: passed", ...]`). String entries matching the strict pattern `^<name>[:\-‚ÄĒ‚Äď]\s*(pass|passed|fail|failed|true|false|yes|no)$` are rescued into the object schema; entries that don't match are logged and discarded. If every entry was a non-dict and none could be rescued, `JudgeExtractionError` is raised to trigger the existing 5-attempt retry loop instead of returning an all-fail map. `build_judge_prompt` now injects concrete rubric dimension names plus an explicit WRONG/RIGHT format block, removing the `"ExactDimensionName"` placeholder that small judges could mistake for a literal value.
- **Judge response parser is now a Postel layer.** `parse_rubric_verdict` accepts five envelope shapes any LLM family can emit, collapsing the `JudgeContractError` retry-storm that drove B06 / B21 latency:
  - `{"dimensions": [...]}` ‚ÄĒ the strict contract (unchanged).
  - `{"verdicts": [...]}` / `{"results": [...]}` / `{"scores": [...]}` / `{"evaluations": [...]}` ‚ÄĒ common wrapper-key aliases, mapped via `_DIMENSIONS_KEY_ALIASES`.
  - `[{"name": ..., "passed": ..., "reasoning": ...}, ...]` ‚ÄĒ bare list at top level (judge dropped the envelope).
  - `{"dim_a": true, "dim_b": false, ...}` ‚ÄĒ flat top-level booleans (existing).
  - `{"dim_a": {"passed": ..., "reasoning": ...}, ...}` ‚ÄĒ flat top-level dicts (existing).
  - `{"dim_a": "passed", "dim_b": "failed", ...}` ‚ÄĒ **new**: flat top-level string verdicts. Tokens accepted: `pass / passed / true / yes / ok` (truthy) and `fail / failed / false / no` (falsy). Unknown tokens still raise `JudgeContractError` ‚Üí retry loop fires as last resort.
  Garbage names still flow through the strict `_fuzzy_match_dim` rubric filter, so the Postel layer adds no false-positive risk. The prompt itself stays strict; only the parser is generous (`_resolve_dimensions_payload` dispatch).
- **Prompt simplification: dropped literal WRONG list example from `build_judge_prompt`.** The H10 WRONG block emitted a stringified `["dim_a: passed", "dim_b: failed"]` counter-example. Instruction-following models with limited capacity could mirror it. The server-side `_rescue_string_entry` guard already handles string-list outputs without the prompt-side negative example. Replaced with a single descriptive constraint sentence (`"Do not return a list of strings ‚ÄĒ every entry must be an object..."`). The positive RIGHT example with concrete rubric dimension names stays ‚ÄĒ it is the teaching signal that works.
- **B29 ‚ÄĒ `derive_secondary_seed` `AttributeError` fix.** Pre-existing bug in `ifixai/harness/prompt_pool.py` since commit `86b2f68`: the function called `random.Random(seed).randbelow(2**31)`, but `randbelow` lives on `secrets.SystemRandom`, not on `random.Random`. Every B29 run errored. Replaced with the standard-library equivalent `random.Random(seed).randrange(2**31)` ‚ÄĒ deterministic from seed, returns `0 ‚Č§ n < 2**31`. Three new tests in `tests/unittesting/harness/test_prompt_pool.py::TestDeriveSecondarySeed` lock the contract (determinism, range, input-decoupling).
- **Provider empty-content distinguishes from configuration failure (B30 fix).** New `ProviderEmptyContentError(ProviderResponseError)` in `ifixai/providers/base.py`. All eight providers (OpenAI, Azure, Anthropic, HuggingFace, OpenRouter, LiteLLM, Bedrock, Gemini) now raise this typed subclass when the upstream API returns a successful response with no text content (safety filter, refusal-as-empty, upstream truncation). `BaseTest.execute` catches it ahead of the generic `except Exception` and maps to `TestStatus.INCONCLUSIVE` with `insufficient_evidence=True` and the provider reason in `error`. **Rationale:** the SUT *ran* ‚ÄĒ it just produced no scoreable output. That is unscorable, not misconfigured. ERROR stays reserved for genuine configuration failures (per H9). `ProviderEmptyContentError` inherits from `ProviderResponseError`, so any existing `except ProviderResponseError` keeps working ‚ÄĒ additive widening only. `execute_conversation` also re-raises the typed exception alongside `JudgePipelineRequiredError`.
- **B05 latency-fix bundle.** Five forward-direction optimizations to the atomic-claims judge path used by B05 (and ground-truth path used by B07):
  - **Parallel user fan-out in B05.** `B05SourceProvenance.run` previously sequenced one user at a time (`for user in fixture.users: await ...`). Now bounded-parallel via `asyncio.Semaphore` (`b05` concurrency limit added to `DEFAULT_INSPECTION_CONCURRENCY` at 12, matching B07). Wall-time drops from O(N) to O(N/concurrency). Mirrors B07's pattern; no semantic change to scoring.
  - **Atomic judge calls now enforce a 60 s per-call timeout** via `asyncio.wait_for(..., timeout=_ATOMIC_JUDGE_TIMEOUT)` in `_send_atomic`. Previously the path inherited only the provider's HTTP timeout (httpx default ‚Čą 5 min), so a hung upstream could stall an entire benchmark via the new sequential-by-design behaviour.
  - **`max_tokens=1024` cap on every atomic judge call** via `_provider_config.model_copy(update={"max_tokens": _ATOMIC_MAX_TOKENS})`. Caps generation cost on verbose fixtures while leaving headroom for ~15 atomic claims at typical verbosity. The original `_provider_config` is not mutated ‚ÄĒ copy-on-call semantics.
  - **Bounded retry (`_ATOMIC_MAX_ATTEMPTS = 2`)** in `_score_with_single_evaluator` and the single-judge branch of `score_atomic_claims_with_ground_truth`. Recovers from transient parse failures and one-off provider hiccups without the analytic-judge path's heavier 5-attempt budget. Worst-case latency capped at ~2 √ó `_ATOMIC_JUDGE_TIMEOUT`. Ensemble path keeps its parallel-redundancy strategy (no per-call retry ‚ÄĒ gather provides resilience).
  - **Postel parser for the atomic JSON payload.** New `_extract_claim_list` accepts strict bare list (happy path), markdown-fenced JSON, and wrapper-key aliases (`claims`, `results`, `verdicts`, `scores`, `evaluations`, `dimensions`). Generic across LLM families; mirrors the Phase 1 Postel layer added to the analytic-judge path. Strict prompt stays unchanged.

  Locked by 9 new tests in `tests/unittesting/evaluation/test_atomic_claims_hardening.py` (parser shapes, retry recovery, retry give-up, `max_tokens` cap propagation) and the existing 9 ground-truth tests stay green unchanged. The atomic path now has all the resilience properties the analytic path got from H10/Phase 1, without re-using a single line of analytic-path code.
